## Блок 1. Теория вероятности и логика

### Задание 1. Фермер — 3,99

Считать распределение числа различных видов напрямую тяжело. Вместо этого
раскладываю величину на индикаторы: пусть `X_i = 1`, если вид `i` был увиден хотя
бы раз за день, и `0` иначе. Тогда число различных видов `X = X_1 + … + X_6`, и по
линейности математического ожидания `E[X] = Σ E[X_i]`. Независимость индикаторов
при этом не требуется — именно это и делает приём удобным, потому что события
«вид 1 увиден» и «вид 2 увиден» на самом деле зависимы.

Каждый из 6 заходов независимо показывает случайный вид из шести. Вероятность,
что конкретный вид не попался ни разу:

    P(X_i = 0) = (5/6)^6 = 15625/46656 ≈ 0,3349
    E[X_i] = P(X_i = 1) = 1 − (5/6)^6 ≈ 0,6651

Отсюда

    E[X] = 6 · (1 − (5/6)^6) = 31031/7776 ≈ 3,9906

**Ответ: 3,99**

Это задача о покрытии купонов в обратной постановке: не сколько попыток нужно,
чтобы собрать все виды, а сколько видов соберётся за фиксированное число попыток.

### Задание 2. Кулинарное соревнование — 26,8

Снова индикаторы: `Y_k = 1`, если шеф с рангом `k` выиграл оба этапа.

Нумерую шефов по возрастанию мастерства, `k = 1, …, 80`. Шеф ранга `k` сильнее
ровно `k − 1` соперников. При случайной жеребьёвке его соперник равновероятно
любой из оставшихся 79, поэтому

    P(выиграл этап) = (k − 1) / 79

Жеребьёвки двух этапов независимы — по условию пары могут повториться, то есть
второй раунд не помнит первый. Значит

    P(выиграл оба) = ((k − 1) / 79)²

Суммирую по всем шефам:

    E = Σ (k=1..80) ((k−1)/79)² = (1/79²) · Σ (j=0..79) j²
      = (1/6241) · (79 · 80 · 159 / 6)
      = 167480 / 6241 = 2120/79 ≈ 26,835

**Ответ: 26,8**

Проверка на здравый смысл: доля побед распределена почти равномерно на отрезке
[0, 1], а среднее квадрата равномерной величины равно 1/3, поэтому ожидаемо
около `80/3 ≈ 26,7` — сходится.

### Задание 3. Одинокая дорога — 63,2; 93,3

Ключевой момент — выбор модели. Появление машин на пустынном шоссе описывается
пуассоновским потоком: события независимы, интенсивность постоянна, а вероятность
появления зависит только от длины интервала, но не от его положения на оси
времени. Тогда вероятность «ни одной машины за время t» равна `e^(−λt)`.

Важно не решать задачу пропорцией: `0,95 · (10/30) = 0,317` — неверно, вероятности
не масштабируются линейно.

Из условия для 30 минут:

    1 − e^(−30λ) = 0,95   →   e^(−30λ) = 0,05

Для произвольного интервала t минут:

    P(t) = 1 − e^(−λt) = 1 − (e^(−30λ))^(t/30) = 1 − 0,05^(t/30)

    t = 10:  P = 1 − 0,05^(1/3) = 1 − 0,3684 = 0,6316  →  63,2 %
    t = 27:  P = 1 − 0,05^(0,9) = 1 − 0,0675 = 0,9325  →  93,3 %

**Ответ: 63,2; 93,3**

## Блок 2. Python

### Задание 1. Изоморфизмы — время O(n), память O(1)

Изоморфность — это биекция между алфавитами двух строк, поэтому одного словаря
`s → t` недостаточно. Проверка только в одну сторону признала бы изоморфными
`'badc'` и `'baba'`: каждый символ первой строки отображается однозначно, но `b` и
`d` при этом схлопываются в один и тот же символ. Поэтому веду два словаря и
проверяю соответствие в обе стороны.

Один проход по строке, операции со словарём в среднем O(1) — **время O(n)**.
Словари хранят не больше символов, чем есть в алфавите — **память O(k)**, а для
фиксированного алфавита это O(1).

Нижняя граница по времени — O(n): не прочитав всю строку, ответить нельзя.
Решение оптимально.

### Задание 2. Натуральная последовательность — время O(n), память O(1)

Сумма полной последовательности 1..n известна заранее и равна `n(n+1)/2`, а из
длины входа знаю `n = len(nums) + 1`. Разность ожидаемой и фактической сумм и есть
пропущенное число. Один проход, две переменные.

Про переполнение: в Python тип `int` неограничен, поэтому суммирование безопасно.
В C++ или Java при больших n сумма может переполнить 32-битный тип, и там надёжнее
вариант через XOR — та же сложность O(n) и O(1), но без роста промежуточных
величин.

**Если список отсортирован** — а в примере из условия это так — задача решается
быстрее. До пропуска выполняется `nums[i] == i + 1`, после пропуска
`nums[i] > i + 1`. Массив разделён на два блока, и границу находит бинарный поиск:
**время O(log n), память O(1)**. Это и есть оптимальное решение; линейный проход в
отсортированном случае избыточен. Обе реализации приведены в файле.

### Задание 3. Факторизация — время O(√n), память O(log n)

Метод пробного деления. Обоснование границы перебора: если n составное, у него
обязательно есть делитель не больше √n — иначе произведение двух его множителей
превысило бы n. Значит перебирать делители дальше √n бессмысленно, и всё, что
осталось больше единицы после цикла, само является простым.

Оптимизация: сначала отдельно вычёрпываю все двойки, затем иду шагом 2 по
нечётным — это вдвое сокращает число итераций.

**Время O(√n)** в худшем случае, когда n простое. **Память O(log n)** — столько
множителей максимум в разложении, потому что минимальный множитель равен 2 и их
не больше `log₂ n`.

Для криптографически больших n пробное деление непрактично; там применяют
ро-алгоритм Полларда со сложностью порядка O(n^(1/4)).

### Задание 1. Абитуриенты

```sql
SELECT id,
       scores,
       RANK() OVER (ORDER BY scores DESC) AS rating_position
FROM examination;
```

Существенный момент — обработка одинаковых баллов. Для баллов 95, 90, 90, 85:

| Функция | Результат | Трактовка |
|---|---|---|
| `RANK()` | 1, 2, 2, 4 | делят второе место, третьего нет |
| `DENSE_RANK()` | 1, 2, 2, 3 | делят второе, следующий — третий |
| `ROW_NUMBER()` | 1, 2, 3, 4 | разрывает ничью произвольно |

Для вступительного рейтинга корректен `RANK()`: равные баллы обязаны получить
равную позицию, а идущий за ними абитуриент не может быть вторым, если двое уже
впереди него. `ROW_NUMBER()` здесь ошибка — он присвоил бы одинаковым результатам
разные места по случайному признаку.

### Задание 2. FULL JOIN — минимально 30 и максимально 600 строк

**Минимум 30.** `FULL JOIN` гарантирует, что каждая строка обеих таблиц попадёт в
результат хотя бы один раз, поэтому результат не может быть короче большей
таблицы. Ровно 30 достигается, когда все 20 ключей второй таблицы уникальны и
каждый находит ровно одно совпадение в первой: 20 сматченных пар плюс 10
оставшихся строк первой таблицы с NULL-ами.

**Максимум 600.** Интуиция подсказывает, что максимум даёт полное отсутствие
совпадений, но это даёт всего 30 + 20 = 50. Настоящий максимум даёт обратное —
полное совпадение при дублирующихся ключах. Если во всех 30 строках первой таблицы
ключ один и тот же и во всех 20 строках второй он же, соединение вырождается в
декартово произведение: 30 × 20 = 600.

### Задание 3. Покупки

```sql
SELECT a.client_id
FROM account a
LEFT JOIN transaction t
       ON t.account_id = a.id
      AND t.type = 'PUR'
      AND t.transaction_date >= DATE_TRUNC('month', CURRENT_DATE) - INTERVAL '1 month'
      AND t.transaction_date <  DATE_TRUNC('month', CURRENT_DATE)
GROUP BY a.client_id
HAVING COALESCE(SUM(t.amount), 0) < 5000;
```

Три содержательных момента.

**Агрегация идёт по клиенту, а не по счёту.** Условие «меньше 5000» относится к
сумме по всем счетам клиента вместе, отсюда `GROUP BY a.client_id`. Фильтр по
агрегату идёт в `HAVING` — в `WHERE` он невозможен, `WHERE` отрабатывает до
группировки.

**JOIN обязан быть LEFT, а условия по дате и типу — в `ON`, не в `WHERE`.** При
`INNER JOIN` из выборки выпадут клиенты, у которых за месяц покупок не было вовсе,
а ноль — это тоже «меньше 5000», и такие клиенты в ответ входят. Перенос тех же
условий в `WHERE` ломает `LEFT JOIN` обратно в `INNER`: подставленные NULL-ы не
пройдут проверку, и строка отбросится.

**COALESCE вокруг SUM.** По пустой группе `SUM` возвращает NULL, а сравнение
`NULL < 5000` даёт NULL, то есть строка снова отбрасывается.

Ограничение «без подзапросов и оконных функций» выполнено: одно соединение,
группировка и `HAVING`.

Допущения, которые в реальной задаче стоит уточнить у постановщика:

- «покупка» принята как `type = 'PUR'` — конкретный код в условии не задан;
- «за последний месяц» прочитано как предыдущий календарный месяц; если имеются в
  виду последние 30 дней, условие меняется на
  `t.transaction_date >= CURRENT_DATE - INTERVAL '30 days'`;
- закрытые счета учитываются, поскольку транзакции по ним клиент всё же совершал;
  если нужны только действующие, добавляется `a.close_dt IS NULL`.

## Блок 4. Статистика и АБ-тесты

### Задание 1. Воодушевленное руководство — верны утверждения 2, 3 и 4

**Неверно:** «Делаем, только применяем изменения в таком же масштабе». Проблема не
в масштабе внедрения, а в том, что выборка не представляет генеральную
совокупность, на которую руководство хочет распространить вывод.

**Верно:** «Тест нерепрезентативен, поэтому результаты применять по всей стране
нельзя». Эксперимент прошёл в одном небольшом городе, а эффект смены транспортного
средства критически зависит от условий: плотность застройки, трафик, длина
маршрутов, климат, качество дорог. В мегаполисе результат может оказаться не
просто слабее, а противоположным.

**Верно:** «Тест репрезентативен относительно своей генеральной совокупности».
Выводы законны для той популяции, из которой бралась выборка, — похожих небольших
городов.

**Верно:** «Наличие эффекта подтвердило потенциал идеи, поэтому сразу применять по
всем городам не будем, но можем провести эксперимент в других городах». Это
описывает корректный следующий шаг: раскатка по стране требует повторения
эксперимента на других типах городов.

Отдельно стоит обратить внимание на формулировку «время доставки значимо снизилось
**в несколько раз**». Эффект такого размера на реальной операционной метрике почти
всегда означает не гениальную идею, а ошибку: сломанный сплитователь, разное
качество логирования в группах, попадание в тест только части заказов, сезонность,
совпавшая с раскаткой. Перед любыми выводами имеет смысл прогнать A/A-тест и
проверить корректность разбиения.

### Задание 2. Основной показатель в статистике

**Верно: «Вероятность наблюдения такого или более экстремального результата при
условии, что нулевая гипотеза верна».**

Это определение по построению: `p = P(данные | H₀)`.

Почему остальные варианты неверны:

- «Вероятность, что нулевая гипотеза верна» — самая частая ошибка. Это
  `P(H₀ | данные)`, то есть перевёрнутое условие. Частотный подход вообще не
  приписывает гипотезе вероятность: H₀ либо верна, либо нет.
- «Значение уровня значимости, при котором отвергается H₀» — путаница с α. Уровень
  значимости задаётся исследователем заранее, до эксперимента, и не зависит от
  данных; p-value вычисляется из данных.
- «Среднее значение выборки» — не имеет отношения к определению.

### Задание 3. Параметрический тест — да, можно

Условие нормальности в t-критерии относится не к распределению самих наблюдений, а
к **распределению выборочного среднего**. По центральной предельной теореме среднее
сходится к нормальному при росте n для любого распределения с конечной дисперсией —
а у логнормального она конечна.

При n больше 5 млн в каждой группе и отсутствии значимых выбросов сходимость
заведомо достигнута. Логнормальное распределение скошено вправо, из-за чего
сходимость медленнее, чем у симметричных, но речь идёт о десятках и сотнях
наблюдений, а не о миллионах.

Что стоит добавить к ответу:

- использовать **t-критерий Уэлча**, а не классический: он не требует равенства
  дисперсий, а в АБ-тестах воздействие часто меняет и разброс метрики;
- проверить **единицу рандомизации**. Метрика измеряется на пользователя, и если
  рандомизация тоже пользовательская, наблюдения независимы. Если бы рандомизация
  шла по сессиям или заказам, независимость нарушилась бы и дисперсия занизилась —
  это была бы более серьёзная проблема, чем форма распределения;
- **не логарифмировать метрику**. Это выглядит естественным при логнормальном
  распределении, но t-критерий на логарифмах проверяет равенство геометрических
  средних, а не арифметических. Целевая метрика — среднее число уникальных покупок,
  поэтому подмена гипотезы недопустима;
- при сомнениях подстраховаться бутстрэпом среднего: на такой выборке он
  практически совпадёт с t-критерием, что и будет подтверждением корректности.

## Блок 5. ML Base

### Задание 1. Пони тоже кони

**Ответ: вторую — инвертировать прогнозы (`p' = 1 − p`) и получить ROC-AUC 0,9,
предварительно убедившись, что причина в перепутанных метках классов, и
перепроверив результат на отложенной выборке.**

ROC-AUC 0,5 соответствует случайному угадыванию. Метрика 0,1 не означает, что
модель плохая, — она означает, что модель уверенно ошибается, то есть содержит
сильный сигнал с обратным знаком. ROC-AUC симметрична относительно 0,5, поэтому
инвертирование прогнозов даёт `1 − 0,1 = 0,9`.

Практически это почти всегда следствие того, что при обучении классы были помечены
наоборот. Первая модель с ROC-AUC 0,7 после такой правки оказывается заметно
слабее второй.

Что сделать до того, как брать модель в работу: убедиться, что 0,1 воспроизводится
на отложенной выборке и не является артефактом протечки или ошибки в разметке наших
данных; проверить, что метки не перепутаны уже у нас, а не в модели; после
инвертирования перепроверить метрику на новых данных.

### Задание 2. Ручной счёт ROC-AUC — 0,75

Порог классификации 0,6 в расчёте не участвует: ROC-AUC оценивает качество
ранжирования по всем порогам сразу, а не при одном конкретном. Колонка с метками
при пороге 0,6 в этой задаче — отвлекающая.

Использую интерпретацию ROC-AUC как вероятности того, что случайно взятый объект
класса 1 получит оценку выше, чем случайно взятый объект класса 0 (эквивалентно
статистике Манна–Уитни). Тогда достаточно посчитать долю правильно упорядоченных
пар «позитив–негатив».

Ход решения:

1. В выборке 7 объектов класса 1 с оценками 0,95; 0,85; 0,75; 0,70; 0,65; 0,60;
   0,40 и 8 объектов класса 0 с оценками 0,90; 0,80; 0,55; 0,50; 0,45; 0,35; 0,30;
   0,25.
2. Всего пар «позитив–негатив»: 7 × 8 = 56.
3. Для каждого позитива считаю, сколько негативов имеют строго меньшую оценку.
   Совпадающих оценок нет, поэтому поправка на связи (по 0,5 за пару) не нужна.

| Оценка позитива | Негативов ниже |
|---|---|

### Задание 3. Ручной счёт корреляции — −0,85

Считаю по формуле

    r = Σ(xᵢ − x̄)(yᵢ − ȳ) / √( Σ(xᵢ − x̄)² · Σ(yᵢ − ȳ)² )

При n = 10: `Σx = 23`, отсюда `x̄ = 2,3`; `Σy = 732`, отсюда `ȳ = 73,2`.

| Кофе | Балл | x − x̄ | y − ȳ | произведение |
|---|---|---|---|---|
| 1 | 85 | −1,3 | 11,8 | −15,34 |
| 1 | 88 | −1,3 | 14,8 | −19,24 |
| 2 | 79 | −0,3 | 5,8 | −1,74 |
| 2 | 81 | −0,3 | 7,8 | −2,34 |
| 2 | 84 | −0,3 | 10,8 | −3,24 |
| 2 | 65 | −0,3 | −8,2 | 2,46 |
| 3 | 67 | 0,7 | −6,2 | −4,34 |
| 3 | 58 | 0,7 | −15,2 | −10,64 |
| 3 | 76 | 0,7 | 2,8 | 1,96 |
| 4 | 49 | 1,7 | −24,2 | −41,14 |

    Σ(x − x̄)(y − ȳ) = −93,60
    Σ(x − x̄)²       =   8,10
    Σ(y − ȳ)²       = 1499,60

    r = −93,60 / √(8,10 · 1499,60) = −93,60 / 110,21 = −0,8493

**Ответ: −0,85**

**Вывод.** Между числом выпитых чашек кофе и баллом за экзамен наблюдается сильная
отрицательная линейная связь: модуль коэффициента около 0,85, коэффициент
детерминации `r² ≈ 0,72` — вариацией числа чашек объясняется около 72 % вариации
балла.

**Причинно-следственную связь утверждать нельзя**, и тому есть три отдельные
причины.

*Корреляция не устанавливает причинность в принципе.* Наблюдательные данные не
позволяют отличить «кофе снижает балл» от обратного направления или от общей
причины.

*Здесь хорошо видна правдоподобная третья переменная — неподготовленность.*
Студент, который плохо знает материал, не спал ночь перед экзаменом: отсюда и
много кофе, и низкий балл. Кофе тогда не причина низкого балла, а маркер недосыпа,
то есть классический конфаундер. Возможно и обратное направление: тревожность перед
трудным экзаменом одновременно повышает потребление кофе и мешает сосредоточиться.

*Выборка слишком мала для сильных заявлений.* При n = 10 оценка коэффициента крайне
неустойчива: одно наблюдение — студент с 4 чашками и баллом 49 — заметно тянет
результат вниз, а доверительный интервал очень широк.

Чтобы говорить о причинности, нужен рандомизированный эксперимент: случайно
назначить студентам разное количество кофе и сравнить баллы. Только случайное
назначение уравнивает группы по всем скрытым факторам, включая подготовку и сон.
